# 제품 컨셉

## 한 줄 정의

개인이 자신의 자산·소득·가족구성 등 정보를 입력하면, 그에 맞는 임대/공공청약
정보를 얻을 수 있는 **개인 대상** MCP 서버.

## 대상 사용자와 상호작용 방식

- 대상은 기관이 아니라 **개인**(예비 청약자/임차인).
- 사용자는 MCP 클라이언트(카카오 LLM 등)를 통해 자연어로 접근한다. 이 서버 자체는
  자연어를 파싱하지 않고, 클라이언트가 구조화해서 넘긴 값(연봉, 가구원수, 결혼상태
  등)만 처리한다 — 설계 원칙 상세는 [../README.md](../README.md) 상단 참고.

## 목표 범위: 임대 + 공공청약 (2개 도메인)

| 도메인 | 상태 | 비고 |
|---|---|---|
| 청약(분양) | 구현됨 — 5개 도구 | 상세는 README [`제공하는 도구`](../README.md#제공하는-도구) 참고 |
| 임대 | 미구현 | 데이터소스·자격기준·도구 구조 전부 미정 |

## 현재 구현 범위와의 간극

- 서비스 소개 문구(README 상단, `server.py`의 `instructions`)는 이미
  "청약(분양·임대)"라고 되어 있지만, 실제 5개 도구는 전부 **분양 청약** 쪽이다.
  지금은 결함이 아니라 **알려진 확장 계획**으로 취급한다.
- 임대까지 구현하고 나면 소개 문구가 실제 범위와 다시 맞는지 재검토한다.

## 임대 도메인 확장 시 미리 알아둘 점 (설계 착수 전 밑그림)

- 임대 관련 공공데이터 API가 무엇인지 조사 필요 (LH 임대주택, 매입임대, 전세임대
  등 — `clients/lh.py`에 이미 있는 오퍼레이션 중 활용 가능한 것부터 검토).
- `eligibility_rules.yaml`의 청약 자격판정 구조를 임대에도 재사용할지, 별도 구조가
  필요한지 미정.
- 도구를 세분화할지 단일 flow-tool로 만들지는
  [architecture-decisions.md의 ADR-002](architecture-decisions.md#adr-002)에
  종속된 결정이다 — 임대 도구를 설계하기 전에 먼저 확인할 것.
